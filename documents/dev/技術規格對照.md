# Kotobase — 技術規格對照

> 整理目前網頁版的技術棧，對照 iOS 手機版的建議方案，以及正式上線的承載力評估。

---

## 網頁版現行技術棧

| 層級 | 技術 | 版本/備註 |
|------|------|----------|
| 前端框架 | React + Vite | React 18.3、Vite 6.4 |
| UI 樣式 | Tailwind CSS | 4.0 |
| 本機儲存 | IndexedDB | kotobase-data（words、categories 兩個 store） |
| 音訊快取 | IndexedDB | kotobase-audio-cache（audio、word-audio 兩個 store，上限 500 條） |
| 偏好設定 | localStorage | 語言、TTS 聲音、TTS prompt |
| 認證 | Firebase Auth | Google OAuth（signInWithPopup） |
| 雲端資料庫 | Firestore | users/{uid} 主文件 + words 子集合 |
| 後端 API | Google Cloud Run | TTS 端點、Furigana 端點 |
| TTS 服務 | Gemini 2.5 Flash + Google Cloud Wavenet | 兩種引擎並存 |
| 平假名標注 | Yahoo Japan ふりがな API | 透過 Cloud Run 代理 |
| 國際化 | i18next | 繁中、日文、韓文（之後加英文） |
| 部署 | GitHub Pages | GitHub Actions 自動部署 |
| 圖標 | lucide-react | 0.395 |
| 彈出視窗 | SweetAlert2 | 11.11 |

## iOS 手機版建議技術棧

| 層級 | 建議技術 | 取代網頁版的 | 說明 |
|------|---------|------------|------|
| 前端框架 | SwiftUI | React | iOS 原生宣告式 UI |
| 本機儲存 | SwiftData | IndexedDB | Apple 原生 ORM，Xcode 15+ 支援，比 Core Data 更簡潔 |
| 音訊快取 | FileManager | IndexedDB | 音檔直接存本機檔案系統，管理更直覺 |
| 偏好設定 | UserDefaults | localStorage | iOS 標準輕量偏好儲存 |
| 認證 | Firebase Auth iOS SDK | Firebase Auth Web SDK | 同一個 Firebase 專案；也可額外支援 Sign in with Apple |
| 雲端資料庫 | Firestore iOS SDK | Firestore Web SDK | 直接連線，與網頁版共用同一個 users/{uid} |
| 後端 API | **同網頁版 Cloud Run** | — | 共用，不需要改動 |
| TTS 服務 | **同網頁版 API** | — | 共用，不需要改動 |
| 平假名標注 | **同網頁版 API** | — | 也可考慮 iOS 原生 CFStringTokenizer |
| OCR 文字辨識 | Vision framework / VisionKit | 網頁版無此功能 | iOS 原生，新功能 |
| 國際化 | String Catalogs | i18next | Xcode 15+ 原生方案 |
| 音訊播放 | AVFoundation | Web Audio API | iOS 原生音訊框架 |
| 部署 | App Store / TestFlight | GitHub Pages | Apple 官方發布管道 |

## 共用後端（不需要改動）

iOS 和網頁版可以完全共用以下後端服務：

| 服務 | 說明 |
|------|------|
| Cloud Run API | TTS（Gemini + Wavenet）和 Furigana 端點，iOS 直接呼叫同一個 URL |
| Firestore | 資料結構不變，同一個 users/{uid}，兩端資料互通 |
| Firebase Auth | 同一個 Firebase 專案，iOS 加上 GoogleSignIn SDK 即可 |

**API 認證方式不變：** iOS 取得 Firebase ID Token 後，以 `Authorization: Bearer {token}` 呼叫 Cloud Run API。

## 本機儲存對照

| 資料 | 網頁版 | iOS 版 |
|------|--------|--------|
| 筆記（words） | IndexedDB words store | SwiftData Word model |
| 分類（categories） | IndexedDB categories store | SwiftData Category model |
| 音訊快取 | IndexedDB audio store（500 條上限） | FileManager 檔案快取（設定上限） |
| 單字綁定音檔 | IndexedDB word-audio store | FileManager + SwiftData 關聯 |
| 語言偏好 | localStorage `appLanguage` | UserDefaults |
| TTS 聲音偏好 | localStorage `ttsVoice` | UserDefaults |
| TTS prompt 偏好 | localStorage `ttsPrompt` | UserDefaults |

## Firestore 資料結構（兩端共用）

```
users/{uid}
├── categories: { [catId]: { label, subcats, customColor } }
├── updatedAt: Timestamp
└── words/（子集合）
    └── {wordId}
        ├── id: string
        ├── title: string
        ├── jp_content: string
        ├── en_content: string
        ├── note: string
        ├── category: string
        ├── subcategories: string[]
        └── created_at: ISO 8601
```

## 正式上線承載力評估

### 目前架構能撐多少？

| 規模 | 使用者數 | 評估 |
|------|---------|------|
| 小規模 | < 1,000 | **完全夠用**，免費方案即可 |
| 中規模 | 1,000 ~ 10,000 | 需升級 Firebase Blaze 方案，監控 API 配額 |
| 大規模 | > 10,000 | 需加入快取層和 CDN，重新設計部分架構 |

### 各服務承載力

| 服務 | 免費方案限制 | Blaze 方案 | 瓶頸風險 |
|------|------------|-----------|---------|
| **Firestore** | 50,000 讀/天、20,000 寫/天 | 按量計費，自動擴展 | 低（擴展性好） |
| **Cloud Run** | 200 萬次請求/月 | 按量計費，自動擴縮容器 | 中（冷啟動 1-2 秒） |
| **Gemini TTS** | 有 rate limit（X-RateLimit-Remaining） | 需申請提高配額 | **高**（大量使用者同時生成語音） |
| **Wavenet TTS** | 透過 Cloud Run 轉發 | 按字數計費 | 中 |
| **Yahoo Furigana** | 免費 API，有呼叫限制 | 無付費方案 | **高**（無法提高限制） |
| **Firebase Auth** | 無限制 | 無限制 | 低 |
| **GitHub Pages** | 100GB 頻寬/月 | — | 低（靜態網頁） |

### 上線前建議優化

| 優先級 | 項目 | 說明 |
|--------|------|------|
| 高 | Cloud Run 設定 min-instances=1 | 避免冷啟動延遲 |
| 高 | 監控 Gemini TTS 配額 | 設定用量警報，接近上限時通知 |
| 中 | Yahoo Furigana 替代方案 | 大量使用時考慮換成自建（MeCab + 辭典）或 iOS 原生 CFStringTokenizer |
| 中 | 升級 Firebase 為 Blaze 方案 | 按量計費，避免觸及免費上限後服務中斷 |
| 低 | 加入 Redis 快取層 | 中大規模時減少 Firestore 讀取和 TTS 重複請求 |
| 低 | 靜態資源上 CDN | 大規模時提升全球載入速度 |

## iOS SPM 套件建議

### 必裝

| 套件 | SPM 來源 | 用途 | 對應網頁版 |
|------|---------|------|-----------|
| **firebase-ios-sdk** | `github.com/firebase/firebase-ios-sdk` | FirebaseAuth + FirebaseFirestore | firebase npm 套件 |
| **GoogleSignIn-iOS** | `github.com/google/GoogleSignIn-iOS` | Google 帳號登入 UI 和流程 | Firebase signInWithPopup |

### 建議安裝

| 套件 | SPM 來源 | 用途 | 說明 |
|------|---------|------|------|
| **Kingfisher** | `github.com/onevcat/Kingfisher` | 圖片載入和快取 | 用於 Google 帳號頭像的非同步載入與快取，避免自己處理 URLSession 圖片下載 |

### 不需要安裝

| 套件 | 原因 |
|------|------|
| **Realm** | SwiftData 已足夠，是 Apple 原生方案，與 SwiftUI 整合最好；此專案資料結構簡單，不需要 Realm 的跨平台能力 |
| **RxSwift** | SwiftUI 內建 Combine 框架 + Swift async/await 已完全取代 RxSwift 的角色；此專案沒有複雜的響應式流需求 |
| **Alamofire** | URLSession + async/await 已經很簡潔，此專案只有 4 個 API 端點，不需要額外的網路層套件 |
| **SnapKit / 其他 Auto Layout** | 使用 SwiftUI，不需要 Auto Layout 工具 |
| **SwiftyJSON** | Swift 原生 Codable 足夠處理 JSON 編解碼 |

### SPM 安裝總結

```
實際需要安裝的只有 3 個套件：

1. firebase-ios-sdk     （必裝 — 認證 + 資料庫）
2. GoogleSignIn-iOS     （必裝 — Google 登入）
3. Kingfisher           （建議 — 頭像圖片載入）

其餘全部用 Apple 原生方案：
- 資料儲存 → SwiftData
- 網路請求 → URLSession + async/await
- 響應式 → Combine / @Observable
- JSON 解析 → Codable
- 音訊播放 → AVFoundation
- OCR → Vision / VisionKit
- 國際化 → String Catalogs
```
